# Grammar Induction

Grammar describes how elements combine to produce an experience. It sits between doctrine and individual outputs.

## Grammar Layers

| Layer | Examples |
|---|---|
| composition | spatial hierarchy, grouping, overlap, contrast, density |
| narrative | setup, reveal, proof, reversal, resolution, action |
| language | sentence rhythm, certainty, analogy, address, evidence placement |
| visual | scale, color roles, typography, imagery, motion, transition |
| sonic | voice, music, silence, emphasis, spatial placement |
| interaction | state, affordance, feedback, navigation, recovery |
| component | reusable sections, shots, controls, scenes, modules |

## Extraction

1. Align examples by production function.
2. Identify repeated relations, not just repeated objects or wording.
3. Record what changes with audience, medium, objective, or tool affordance.
4. Separate genre convention from system-specific deviation.
5. Define parameters, constraints, selection rules, and invalid combinations.
6. Test a new composition that preserves relations while changing expression.

## Tool-Conditioned Grammar

The source of truth changes the possible grammar. Code-native systems favor parameterization, reusable components, deterministic variants, and diffable change. Timeline-native systems favor clip-level context, visual iteration, and human-agent shared state. Record these as constraints or affordances, not as universal quality claims.

## Pattern Record

Use Markdown for rationale and examples, plus a structured pattern when a compiler handoff needs stable fields. A grammar pattern must contain:

- function and intended experience
- eligible contexts
- elements and required relations
- parameters and constraints
- selection rules
- anti-patterns
- medium and tool assumptions
- provenance, confidence, and counterexamples

Do not promote a grammar from one example beyond `candidate` status.
