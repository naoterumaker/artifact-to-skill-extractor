# Self-Extraction Audit

## Artifact And Evidence

The input artifact is this repository: `SKILL.md`, references, templates, scripts, tests, worked outputs, evaluation records, and two explicit owner reviews supplied after v0.2.

Repository structure and behavior are direct evidence. The owner's corrections are eligible evidence for this compiler's project doctrine because the owner is defining the intended system. They are not evidence that the same doctrine applies to every future extracted system.

## Recovered Production Function

The repository converts incomplete evidence into a scoped, executable, and testable production system:

```text
capability contract
  -> corpus and evidence ceiling
  -> native observations and comparisons
  -> supported doctrine + grammar + decision units
  -> capabilities + agency + execution graph
  -> frozen evaluation contract
  -> agent-callable package
  -> unseen concrete output
  -> lowest-layer repair
  -> scoped lifecycle decision
```

`SKILL.md` owns activation and orchestration. References own detailed reasoning. Assets own copyable templates and lawful fixtures. Scripts own deterministic checks. Mutable run evidence belongs in an external project workspace.

## v0.2 Audit

The first self-audit found that the Generator/Evaluator loop existed but cross-domain promotion, repeated-failure harvesting, lifecycle demotion, and large-corpus estimation were incomplete. The retrospective pre-repair score was `75.70`; the repaired state scored `88.20`.

That audit added transfer, failure-harvest, lifecycle, cost, estimation, and prompt-test support. Because its rubric was reconstructed after implementation began, the score is diagnostic evidence, not valid promotion evidence.

## v0.3 Owner-Review Findings

Two later reviews exposed a deeper architectural gap:

1. v0.2 still leaned toward extracting patterns and workflows rather than compiling evidence-scoped production knowledge.
2. Function labels were not guaranteed to become executable conditional decisions.
3. One artifact could still be read too confidently as evidence of a creator's process or philosophy.
4. Creative work lacked explicit doctrine, value conflicts, audience relationship, epistemology, grammar, medium materiality, agency, and distribution context.
5. Tools were treated mainly as implementation support rather than capability-constrained parts of the production system.
6. X Article and image examples passed output rubrics but did not expose complete v0.3 intermediate representations.

## v0.3 Repair

The repairs were routed to the compiler architecture rather than patched only into example outputs:

- `references/system-model.md` defines procedural, creative, and hybrid profiles plus production-system layers.
- `references/design-doctrine.md` records the owner's corrections as this compiler's explicit constitution.
- `references/comparative-induction.md` aligns works, revisions, accepted/rejected alternatives, success/failure, and real decision opportunities.
- `references/doctrine-induction.md` separates recovered creator doctrine from project doctrine, constraints, and model preference.
- `references/grammar-induction.md` models conditional composition and expression relations.
- `references/decision-unit-ir.md` compiles knowledge into `WHEN / IF / DO / VERIFY / ELSE` contracts.
- `references/tool-capability-graph.md` and `references/capability-planning.md` make capability, source of truth, state, agency, and recovery explicit.
- `scripts/validate_system_ir.py` checks decision, tool, and graph interfaces before promotion.
- Both worked tests now provide complete creative-system intermediate representations.

## Verification And Limits

The repository skill passes strict validation. The X Article creative system and image hybrid system pass complete IR validation. The test suite verifies all three initialization profiles, rejects unfilled templates as incomplete, and preserves the two existing forward-output score improvements.

This still does not justify `active` status:

- each modality has one forward output
- creator-doctrine induction has no longitudinal corpus or held-out prediction result
- evaluations are not blinded or independent
- live platform and audience outcomes are unavailable
- process, research, code, video, audio, and stateful editing systems remain untested

Self-application is therefore real but scoped: the repository changed in response to evidence about its own missing architecture, while preserving unresolved limitations instead of using self-description as proof.
